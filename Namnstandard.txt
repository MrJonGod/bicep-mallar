Generellt gällande namnsättning

Huvuddelen av alla resurser namnsätts enligt följande mönster:

[kundprefix]-[domän/teknisk area]-[funktion]-([subfunktion])-([löpnummer])-([region])-[miljö]-[resurstyp]

Exempel:
abg-dataplatform-api-dab-t-ci
(AB Gavlegårdarna, dataplattform, API-lagret, DAB, testmiljö, Azure Container Instance)

lejon-int-shared-msgbroker-d-sbns
(Lejonfastigheter, domän=integration, funktion=delad resurs, funktion=message broker, dev, service bus)

abg-shared-api-p-apim
(AB Gavlegårdarna, delade resurser, API-lagret, produktionsmiljö, Azure API Management)

lejon-int-dynamics-d-rg
(Lejonfastigheter, domän=integration, funktion=dynamicsintegrationer, dev, resursgrupp)

telge-int-stratsys-d-appi
(Telge, domän=integration, funktion=stratsysintegrationer, dev, application insight)

Det fulla resursnamnet skrivs med kebab-case. Enskilda komponenter i ett resursnamn skrivs enbart med lowercase. Exempelvis blir då en enskild komponent getorders och inte GetOrders.

    Kundprefix
    är en förkortning för kunden. Detta för att enklare identifiera kundens resurser samt skapa globalt unika namn.
    Exempel: Solution Xperts = sx

    Domän/teknisk area
    Samlar resursgrupper.
    Exempel: finance, dataplatform, int(egration)

    Funktion, subfunktion
    Används för att detaljera resurserna i minsta beståndsdelar. Är ett bredare begrepp än bara "användningsområde": funktionen kan också vara att samla resurser som hör till en integrationsgrupp, som i exemplen ovan.

    Löpnummer
    I de fall man har två resurser som har samma syfte men som kräver flera instanser kan löpnummer användas, använd då tre siffror med start på 001. Bör användas restriktivt. Exempel: om vi har två funktionsappar i samma rg där en är kopplad till vnet och den andra inte, bör de inte heta 001 och 002, utan syftet (vnet) tydliggörs i subfunktionsdelen av namnet för den ena fappen. Samma om man har ett storage account för claim check och ett för arkivering i samma rg så bör inte de heller heta 001 och 002 utan syftet ska framgå i namnet. Ett bra scenario är dock när man splittat storage i 2 olika accounts för att hantera ex. begränsningar i antal requests men från koden ser det som samma storage, då är det lämpligt att benämna dessa som instans 001 och 002 utan att skilja på dem på något annat sätt.

    Region
    Region är inte nödvändigt att specificera annat än för de resurser där region kan ha betydelse.
    Regioners förkortningar hämtas här: Geo-code mapping - Azure Backup | Microsoft Learn .

    Miljö
    Miljö är en bokstav (d, t, q/s, p) i stället för dev/development, test, qa, staging/stg, prod.

    Resurstyp
    Resurstypen förkortas och anges alltid sist i namnet.
    Exakt resurstyp är aldrig en del av domän/funktion i namnet.
    Exempelvis sx-finance-processinvoicefunction-func är fel och borde istället vara sx-finance-processinvoice-func

Resurser med längdbegränsningar i namnsättning

Storage accounts och keyvaults har begränsningar i antal tecken, max 24, och eftersom namnen behöver vara globalt unika kan man lätt få problem med att hitta bra namn här.
Storage accounts namnsätts enligt följande:

[kundprefix][domän/teknisk area][funktion][(löpnr)][miljö]st

Exempel:
lejonintdynamicsdst
lejon int dynamics d st

telgeintstratsyspst
telge int stratsys p st
Keyvaults på samma sätt, men med kv som suffix:

[kundprefix][domän/teknisk area][funktion][(löpnr)][miljö]kv

Exempel:
lejonintdynamicsdkv
lejon int dynamics d kv

telgeintstratsyspkv
telge int stratsys p kv
Motivering bakom namnstandarden

Logik bakom namnsättningen är att följa dels en modell för storleksmässig gruppering av resurser (från vänster till höger), dels placera ”metadata” efter. Domän/teknisk area, funktion och subfunktion grupperar resurser som hör ihop enligt ett logiskt mönster, likartat ett namespace. Region, miljö och resurstyp är metadata som kommer sist.

Vissa namnkomponenter kan utelämnas om de inte fyller något syfte. Det är inte tvunget att huvuddelen av namnet består av de tre delarna domän-funktion-subfunktion.

Alla namnkomponenter som kan ses som ”metadata” hålls så korta som möjligt, exempelvis kundprefix, domän, miljö. Detta för att ge så många tecken och utrymme som möjligt till den del av namnet som beskriver resursens syfte.

Förkortningar för Azure-resurser enligt Microsofts standard: Abbreviation recommendations for Azure resources - Cloud Adoption Framework | Microsoft Learn
(Notera att servicebus-resursers namn INTE får sluta på "sb" pga gamla begränsningar i Azure. Därför ersätts det för vår del med "sbus".)

Förkortning för en specifik region enligt Microsofts lista för backup (enda "officiella listan på förkortningar av regioner där Sverige ingår): Geo-code mapping - Azure Backup | Microsoft Learn
